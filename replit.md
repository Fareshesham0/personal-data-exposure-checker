# Personal Data Exposure Checker (PDEC)

## Overview

Academic Final Year Project (FYP). A privacy-preserving web tool that checks whether an email address or password has appeared in known data breach datasets. No user data is stored or logged — the system is fully stateless.

## Stack

- **Monorepo tool**: pnpm workspaces
- **Node.js version**: 24
- **Package manager**: pnpm
- **TypeScript version**: 5.9
- **Frontend**: React + Vite (`artifacts/exposure-checker`)
- **API framework**: Express 5 (`artifacts/api-server`)
- **Database**: None — app is stateless by design
- **Validation**: Zod (`zod/v4`)
- **API codegen**: Orval (from OpenAPI spec at `lib/api-spec/openapi.yaml`)
- **React Query hooks**: auto-generated by Orval into `lib/api-client-react`
- **Zod schemas**: auto-generated by Orval into `lib/api-zod`
- **Security**: Helmet + express-rate-limit

## Free APIs Used (No API Key Required)

| Service | Purpose | Privacy |
|---|---|---|
| XposedOrNot (`api.xposedornot.com`) | Email breach name lookup | Email is transmitted (normalised) |
| HIBP Public (`haveibeenpwned.com/api/v3/breaches`) | Full breach metadata cache | No identifier transmitted |
| HIBP Pwned Passwords (`api.pwnedpasswords.com`) | Password exposure check | Only SHA-1 prefix (5 chars) sent — k-anonymity |

## Key Commands

- `pnpm run typecheck` — full typecheck across all packages
- `pnpm --filter @workspace/api-spec run codegen` — regenerate API hooks and Zod schemas from OpenAPI spec
- `pnpm --filter @workspace/api-server run dev` — run API server locally
- `pnpm --filter @workspace/api-server run test` — run API unit tests (Node 24 built-in `node:test`, no extra deps)

  routes/stats.ts                  → GET /api/stats — in-memory metrics snapshot, response validated by generated Zod (`GetMetricsResponse`).   lib/hibp.ts                      → XposedOrNot + HIBP free API calls (instrumented for external-call latency + cache hit/miss)

## Password Strength Analysis (zxcvbn)

- The Password tab on the home page shows a live, debounced (150 ms) strength meter as the user types, powered by `@zxcvbn-ts/core` + `@zxcvbn-ts/language-en`. The password never leaves the browser for this analysis.
- After a password breach check, the results page re-renders the meter alongside the breach result so users see both signals together. The plaintext password is held in React context memory only and cleared whenever the exposure result is cleared.
- The methodology page documents what the 0–4 zxcvbn score means and that the analysis is local-only.

## Risk Scoring Logic

- **Email overall risk** (0–100): `frequency` (≤40, breach count × 8) + `recency` (≤30, age-bucketed) + `sensitivity` (≤30, avg data-class weight). Buckets: none (0) / low (<30) / medium (<60) / high (≥60). Returned with a `factors` breakdown the UI renders as bars.
- **Password risk** (0–100): single-dimension lookup against HIBP Pwned Passwords occurrence count.
- **Per-breach severity** (0–100): each breach also scored independently (`assessBreachSeverity`) using sensitivity (≤50) + recency (≤30) + scale (≤20, from PwnCount). Surfaced as a per-card badge.
- All formulas, weight tables, the k-anonymity flow, data sources, and academic references (Junade Ali, NIST 800-63B, GDPR Art. 32) live on the public `/methodology` page.
